# 감리·심사 지적사례 DB 앱 — 프로젝트 노트

## 목적
회계감사인이 감사업무 수행 전 유의사항을 파악할 수 있도록, 금융감독원(FSS)·한국공인회계사회(KICPA)의
감리·심사 지적사례를 카테고리별로 검색·열람할 수 있는 웹앱. 사용자는 CPA 합격 후 Big4 Tax 지원을 준비 중이며
실무 감사절차 학습 목적으로 이 도구를 사용.

## 산출물 (파일)
- `audit_review_case_db.html` — 최종 앱 (단일 HTML 파일, 데이터 내장, 브라우저에서 바로 열림)
- `cases_data.json` (Claude Code 쪽에 새로 요청 시 재생성 필요) — 원본 데이터 + 표준화 태그가 적용된 가공 데이터
- 원본 업로드 파일: `accounting_review_cases_db_3.json` (사용자가 직접 학습시킨 원본 DB, 총 94건 상세사례 + KICPA 2027 중점심사 예고 4건)

## 데이터 구조
원본 JSON 최상위 키:
- `meta`: 출처 목록(KICPA/FSS 공시자료), 총 사례 수
- `kicpa_2027_focus_issues`: 2027년 중점심사 예고 이슈 4건 (issue_id, title, standards, background, error_examples, key_points, selection_criteria)
- `fss_cases_2025H2`, `fss_cases_2206`, `fss_cases_2311`, `fss_cases_2512`, `fss_cases_2409`, `fss_cases_2405`, `fss_cases_2505`: 연도별 상세 지적사례 리스트 (총 94건)

각 사례(case) 필드:
```
case_code, category, title, issue_area, standards[], decision_year, accounting_period, summary,
detail: { facts, violation, basis, audit_deficiency(nullable), implication, agent_determination_basis(선택) }
```

## 표준 카테고리 태깅 (앱에서 사용)
원본 `category`/`issue_area` 필드가 너무 세분화되어 있어(1건짜리 태그 다수), 아래 17개 표준 카테고리로 재분류하는
키워드 매칭 로직을 적용함 (파이썬 스크립트로 전처리 → `tags` 필드 추가):

매출채권 / 재고자산 / 유형자산 / 무형자산 / 매출·매출원가·수익인식 / 종속기업투자주식·연결재무제표 /
관계기업투자주식·지분법 / 금융상품·금융자산·파생상품 / 금융부채·전환사채 / 특수관계자거래·공시 /
법인세회계 / 퇴직급여 / 리스 / 충당부채·우발부채 / 주석미기재·재무제표 표시 / 부정회계·허위계상 / 기타

하나의 사례가 여러 이슈에 해당하면 복수 태그 부여. 태깅 규칙은 키워드 기반이라 완벽하지 않을 수 있음(사용자 피드백 반영 필요).

## 앱 기능
1. 카테고리 그리드 (건수 표시) → 클릭 시 해당 카테고리 사례만 필터링
2. 키워드 검색 (회사/계정과목/키워드), 결정연도 필터
3. 사례 카드 클릭 시 펼침: 사실관계 / 위반내용 / 판단근거 / 지적근거(회계기준) / 감사절차 미흡사항 / 실무 시사점 / 관련 기준서
4. 별도 탭: KICPA 2027년 중점심사 예고 4건 (issue_id, 배경, 오류사례, 핵심 포인트, 표본선정기준)

## 디자인 방향
- 공식 규제기관 문서/법률 케이스 데이터베이스 느낌 (미색 종이 배경 + 네이비 잉크 + 붉은 "직인" 스탬프 포인트)
- AI 생성 느낌 나는 기본 팔레트(크림+테라코타, 다크+네온그린 등) 지양
- 가시성 개선 요청 반영: 전체 대비 강화, 카테고리별 고유 색상 코딩(17색 팔레트), 선택 상태를 배경 채움으로 명확히 표시,
  섹션 라벨(사실관계/위반내용 등)을 배지 형태로 강조

## Claude Code에서 이어가는 방법
1. 이 노트 파일과 `audit_review_case_db.html`, 원본 `accounting_review_cases_db_3.json`을 같은 프로젝트 폴더에 둔다.
2. Claude Code 실행 후: "PROJECT_NOTES.md 읽고 이 프로젝트 이어서 작업하자" 라고 시작.
3. 원본 JSON에 새 사례를 추가하고 싶으면, 새 자료(PDF 등)와 원본 JSON을 함께 주고 "스키마에 맞춰 추가해줘"라고 요청.
